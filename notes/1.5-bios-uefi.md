# Phase 1 — Computer Fundamentals
## Topic 1.5 — BIOS/UEFI (How a Computer Starts)

---

## 1. TOPIC OVERVIEW

You press the power button. What happens before Windows or Linux even loads?

**BIOS or UEFI runs first.**

This is the very first software that executes on your machine. Attackers who compromise BIOS have the deepest possible access — **below the operating system.** Understanding this makes you a better penetration tester and security professional.

---

## 2. CONCEPT EXPLANATION

**BIOS** = Basic Input Output System
**UEFI** = Unified Extensible Firmware Interface

Both are **firmware** — meaning software permanently stored on a chip on your motherboard.

> **Firmware** = Software that lives inside hardware chips, not on your hard drive.

Their job is simple:
1. Wake up the hardware when power turns on
2. Check everything is working
3. Find the operating system on storage
4. Hand control over to it

> Simple analogy:
> BIOS/UEFI is like a **security guard at a building entrance.**
> - First checks all doors and cameras are working (hardware check)
> - Then checks your ID (finds the OS bootloader)
> - Then lets you in (hands control to OS)

---

## 3. INTERNAL WORKING

### 🔲 The Boot Process Step by Step

```
Power Button Pressed
        ↓
BIOS/UEFI chip activates
        ↓
POST runs (Power-On Self Test)
        ↓
Hardware check passes
        ↓
BIOS looks for bootable device
        ↓
Finds bootloader on HDD/SSD
        ↓
Loads bootloader into RAM
        ↓
Bootloader loads Operating System
        ↓
OS takes full control
```

---

### 🔲 POST — Power On Self Test

First thing BIOS/UEFI does after power on.

It checks:
- Is RAM working?
- Is CPU working?
- Is storage connected?
- Are keyboard and mouse connected?

If something fails — you hear **beep codes** or see an error screen before OS loads.

---

### 🔲 BIOS vs UEFI — What's the Difference?

| Feature | BIOS | UEFI |
|---|---|---|
| Age | Old (1970s) | Modern (2010s+) |
| Interface | Text only | Graphical, mouse support |
| Drive size support | Max 2TB | 9.4 ZB (unlimited practically) |
| Boot speed | Slower | Faster |
| Security features | Minimal | Secure Boot, encryption |
| Architecture | 16-bit | 32/64-bit |

**Almost all modern computers use UEFI.** BIOS is mostly legacy now.

---

### 🔲 UEFI Secure Boot

A security feature in UEFI that:
- Checks the **digital signature** of the bootloader before running it
- If signature is invalid or missing — boot is blocked
- Prevents malicious bootloaders from loading before the OS

> **Why it matters for hacking:**
> When installing **Kali Linux**, you sometimes need to **disable Secure Boot** because Kali's bootloader may not have a Microsoft-approved signature.
> This is completely normal and legal on your own machine.

---

### 🔲 BIOS/UEFI Settings You Will Use

| Setting | Purpose |
|---|---|
| **Boot Order** | Which device boots first (USB, HDD, network) |
| **Secure Boot** | Enable/disable signature checking |
| **Virtualization (VT-x/AMD-V)** | Must enable for VirtualBox to work |
| **BIOS Password** | Locks BIOS settings from being changed |
| **TPM** | Trusted Platform Module — hardware security chip |

> **Critical for your lab:**
> To run VirtualBox and Kali Linux properly, you **must enable Virtualization** in UEFI settings. This is called **VT-x** (Intel) or **AMD-V** (AMD).

---

### 🔲 How to Access BIOS/UEFI

During startup, press one of these keys rapidly:

| Brand | Key |
|---|---|
| Most common | **F2** or **Delete** |
| HP | F10 |
| Dell | F2 |
| Lenovo | F1 or F2 |
| ASUS | F2 or Delete |
| Acer | F2 |
| Boot menu only | **F12** (most brands) |

---

### 🔲 CMOS and CMOS Battery

BIOS settings are saved in a tiny chip called **CMOS.**

A small battery on the motherboard keeps CMOS powered even when PC is off — this is why your clock stays correct.

**Attacker trick:**
> Removing the CMOS battery resets all BIOS settings including the BIOS password — a physical attack to bypass BIOS security.

---

## 4. WHY IT MATTERS

- **Virtualization must be enabled** in UEFI for your hacking lab to work
- **Secure Boot must sometimes be disabled** to boot Kali Linux
- **Bootkit malware** hides in the boot process — below the OS, extremely dangerous
- **BIOS passwords** are a physical security control
- **Network booting (PXE)** can be exploited in enterprise environments

---

## 5. REAL-WORLD EXAMPLE

**Real Attack — Bootkit Malware**

A **bootkit** is malware that infects the boot process — before the OS even loads.

Famous example: **LoJax (2018)**
- First real-world UEFI rootkit discovered
- Created by APT28 (Russian state-sponsored hackers)
- Survived complete OS reinstallation
- Survived hard drive replacement
- Only way to remove: **flash the UEFI chip** or replace motherboard

This shows why firmware-level attacks are the most dangerous — they sit below everything else.

---

## 6. ATTACKER PERSPECTIVE 🔴

| Attack | How BIOS/UEFI is Involved |
|---|---|
| **Bootkit** | Malware infects boot process, loads before OS |
| **UEFI Rootkit** | Hides malware in UEFI firmware permanently |
| **CMOS Reset** | Physical attacker removes battery to clear BIOS password |
| **Evil Maid Attack** | Attacker with physical access modifies boot process |
| **PXE Boot Attack** | Attacker serves malicious OS image over network boot |
| **Cold Boot Attack** | Boots from USB to access another OS's RAM/storage |

---

## 7. DEFENDER PERSPECTIVE 🔵

| Defense | What It Does |
|---|---|
| **UEFI Secure Boot** | Blocks unsigned bootloaders |
| **BIOS Password** | Prevents unauthorized BIOS changes |
| **TPM Chip** | Hardware security for boot integrity verification |
| **BitLocker + TPM** | Drive encryption tied to boot integrity |
| **Physical Security** | Lock server rooms — CMOS reset needs physical access |
| **UEFI Firmware Updates** | Patches firmware vulnerabilities |

---

## 8. COMMON MISTAKES BEGINNERS MAKE

- Forgetting to **enable Virtualization** in UEFI before setting up VirtualBox
- Not knowing which key opens BIOS on their laptop — just search "[laptop brand] BIOS key"
- Thinking reinstalling Windows removes all malware — bootkits survive reinstallation
- Confusing **BIOS password** with **Windows login password** — completely different

---

## 9. PRACTICAL LAB 🧪

**Objective:** Access your UEFI and enable Virtualization for your future lab

**Steps:**

```
1. Restart your computer

2. Immediately press F2 or Delete rapidly
   (Check your laptop brand's specific key)

3. Navigate to:
   Advanced → CPU Configuration → Intel Virtualization
   OR
   Advanced → SVM Mode (for AMD)

4. Set it to: ENABLED

5. Go to Boot tab
   Check your Boot Order
   Note what's first (HDD, USB, Network)

6. Check if Secure Boot is present
   Location: Security tab or Boot tab
   Note whether it's enabled or disabled

7. Press F10 to Save and Exit
```

**Expected Outcome:**
- Virtualization is now enabled
- Your system is ready for VirtualBox installation
- You now know your boot order

**Key Learning Point:**
You just configured **firmware-level settings** — the same settings attackers target in advanced attacks.

## 10. KEY TERMS

| Term | Simple Meaning |
|---|---|
| **BIOS** | Old firmware that starts your computer |
| **UEFI** | Modern replacement for BIOS, faster and more secure |
| **Firmware** | Software permanently stored inside hardware chips |
| **POST** | Hardware self-check that runs at startup |
| **Bootloader** | Small program that loads the OS from storage |
| **Secure Boot** | UEFI feature that verifies bootloader signatures |
| **Boot Order** | Priority list of devices UEFI tries to boot from |
| **CMOS** | Tiny chip that stores BIOS settings |
| **CMOS Battery** | Keeps CMOS powered when PC is off |
| **Bootkit** | Malware that infects the boot process |
| **UEFI Rootkit** | Malware hidden inside UEFI firmware |
| **PXE Boot** | Booting a computer over the network |
| **TPM** | Hardware chip that verifies system integrity |
| **VT-x / AMD-V** | CPU virtualization features needed for VirtualBox |
