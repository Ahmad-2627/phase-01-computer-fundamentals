# Phase 1 — Computer Fundamentals
## Topic 1.9 — Memory Management

---

## 1. TOPIC OVERVIEW

You know what RAM is and what processes are. Now the question is — **who controls how RAM is shared between all running processes?**

The OS does — through **Memory Management.**

This topic explains how the OS allocates, protects, and reclaims memory. It also explains why some of the most powerful hacking techniques — buffer overflows, heap sprays, use-after-free attacks — exist and how they work conceptually.

---

## 2. CONCEPT EXPLANATION

**Memory Management** is the OS's job of:
- Giving each process the RAM it needs
- Making sure processes don't interfere with each other
- Reclaiming RAM when a process finishes
- Handling situations when RAM runs out

> Simple analogy:
> Think of RAM as a **hotel with limited rooms.**
> - Hotel manager = OS Memory Manager
> - Guests = Processes
> - Rooms = RAM blocks
>
> The manager:
> - Assigns rooms when guests arrive (process starts)
> - Ensures guests don't enter each other's rooms (process isolation)
> - Cleans and reassigns rooms when guests leave (memory deallocation)
> - Uses a waiting area when hotel is full (virtual memory/swap)

---

## 3. INTERNAL WORKING

### 🔲 Memory Allocation

When a process needs memory, it asks the OS.

Two main types of memory allocation:

| Type | Where | Who Controls | Speed |
|---|---|---|---|
| **Stack allocation** | Stack section | OS/Compiler automatic | Very fast |
| **Heap allocation** | Heap section | Programmer manually | Slower |

---

**Stack Memory:**
- Automatically managed
- Fixed size
- Used for function calls and local variables
- Works like a stack of plates — last in, first out (LIFO)

```
Function called → memory pushed onto stack
Function returns → memory automatically popped off stack
```

**Heap Memory:**
- Manually managed by programmer
- Dynamic size — can grow and shrink
- Used when you don't know how much memory you need in advance

```
Programmer requests memory → OS allocates heap space
Programmer must free it when done → OS reclaims it
```

> **Why it matters for hacking:**
> Forgetting to free heap memory = **memory leak**
> Writing too much data into stack = **stack buffer overflow**
> Writing too much data into heap = **heap overflow**

---

### 🔲 Virtual Memory — Every Process Gets Its Own Space

You learned about virtual memory as backup storage. Now let's go deeper.

The OS gives every process its own **virtual address space** — the process thinks it has the entire memory to itself.

```
Process A thinks it has:     Process B thinks it has:
Address 0 to 4GB             Address 0 to 4GB
┌──────────────┐             ┌──────────────┐
│ Its own RAM  │             │ Its own RAM  │
└──────────────┘             └──────────────┘
        ↓                           ↓
   OS translates both to actual physical RAM locations
```

This is called **memory virtualization** and the translation is done by the **MMU (Memory Management Unit)** — a component inside the CPU.

> **Why it matters for hacking:**
> Processes are isolated because they each have separate virtual address spaces. Attacks like **process injection** bypass this by using OS functions to write into another process's virtual space with special permissions.

---

### 🔲 Paging — How Virtual Memory Maps to Physical RAM

RAM is divided into fixed chunks called **pages** (typically 4KB each).

Virtual pages map to physical pages through a **page table.**

```
Virtual Page 1 → Physical Page 7
Virtual Page 2 → Physical Page 2
Virtual Page 3 → DISK (swapped out — not enough RAM)
```

When a process needs a page that's been moved to disk, a **page fault** occurs — OS fetches it back into RAM.

> **Why it matters for hacking:**
> Sensitive data like encryption keys swapped to disk (pagefile/swap) can sometimes be recovered by attackers even after the process ends.

---

### 🔲 Memory Protection — How OS Keeps Processes Isolated

The OS enforces strict rules:

| Protection | Meaning |
|---|---|
| **Read-only pages** | Code sections can't be modified while running |
| **No-execute pages (NX/DEP)** | Data pages can't be executed as code |
| **ASLR** | Randomizes where everything is placed in memory |
| **Stack canaries** | Special values detect stack overflow attempts |
| **Guard pages** | Unmapped pages that trigger errors if accessed |

---

### 🔲 Buffer Overflow — When Memory Management Breaks

A **buffer** is a fixed-size section of memory reserved for data.

A **buffer overflow** happens when more data is written into a buffer than it can hold — overflowing into adjacent memory.

```
Buffer size: 8 bytes
Attacker input: 20 bytes

Before overflow:        After overflow:
┌────────────┬──────┐  ┌────────────┬──────┐
│ Buffer     │ EIP  │  │AAAAAAAAAAAAAAAAAAA│
│ [8 bytes]  │      │  │            │AAAA  │ ← EIP overwritten!
└────────────┴──────┘  └────────────┴──────┘
```

By overwriting **EIP** (remember Topic 1.2), the attacker redirects the CPU to execute their malicious code.

> This is one of the most classic and important attack techniques in penetration testing. You will practice this in advanced phases.

---

### 🔲 Heap Spray

An attacker fills large portions of the heap with malicious code, hoping that when execution is redirected, it lands on their code.

```
Heap memory:
┌────┬────┬────┬────┬────┬────┬────┬────┐
│EVIL│EVIL│EVIL│EVIL│EVIL│EVIL│EVIL│EVIL│
└────┴────┴────┴────┴────┴────┴────┴────┘
↑ Attacker fills heap with malicious code
  High probability redirected execution hits it
```

---

### 🔲 Use-After-Free

Happens when a program:
1. Allocates memory on heap
2. Frees that memory
3. Still uses a pointer to that freed memory
4. Attacker fills that freed space with malicious data
5. Program executes attacker's data as code

> This is a common vulnerability in browsers and complex applications.

---

### 🔲 Memory Forensics

Because sensitive data lives in RAM, investigators dump and analyze it.

**What memory forensics finds:**
- Running malware (even fileless malware)
- Encryption keys
- Passwords in plaintext
- Network connections
- Browser history
- Chat messages

**Tool used:** **Volatility** — the industry standard memory forensics framework

```bash
# Example Volatility command (educational)
volatility -f memory.dump --profile=Win10x64 pslist
# Lists all processes that were running when memory was captured
```

---

## 4. WHY IT MATTERS

- **Buffer overflows** are the foundation of many exploits
- **Heap spray and use-after-free** are common in browser exploitation
- **ASLR and DEP/NX** are defenses you must bypass in advanced pentesting
- **Memory forensics** is used to find fileless malware that never touches disk
- **Paging and swap** can leak sensitive data to disk unexpectedly

---

## 5. REAL-WORLD EXAMPLE

**Scenario — Fileless Malware**

Traditional malware: Drops a file on disk → antivirus scans and detects it

**Fileless malware:**
1. Attacker sends malicious PowerShell command
2. PowerShell loads malicious code **directly into RAM**
3. Malicious code runs entirely in memory
4. **Nothing is ever written to disk**
5. Traditional antivirus finds nothing
6. When system reboots — malware disappears from RAM

**How defenders catch it:**
- **Memory forensics** with Volatility
- **EDR tools** that monitor RAM behavior
- **PowerShell logging** that records commands before they execute

This is why memory management knowledge is critical for both attackers and defenders.

---

## 6. ATTACKER PERSPECTIVE 🔴

| Attack | Memory Management Involvement |
|---|---|
| **Stack buffer overflow** | Overflow stack buffer to overwrite EIP |
| **Heap overflow** | Corrupt heap memory structures |
| **Heap spray** | Fill heap with malicious code |
| **Use-after-free** | Exploit freed memory still being referenced |
| **ASLR bypass** | Find ways to predict randomized memory addresses |
| **Fileless malware** | Run entirely in RAM, never touch disk |
| **Memory scraping** | Read sensitive data from process memory (POS malware) |

---

## 7. DEFENDER PERSPECTIVE 🔵

| Defense | What It Does |
|---|---|
| **DEP/NX** | Marks data pages as non-executable |
| **ASLR** | Randomizes memory layout so attackers can't predict addresses |
| **Stack canaries** | Detects stack overflow before EIP is reached |
| **Safe heap allocators** | Modern heap implementations resist overflow attacks |
| **Memory forensics** | Detects fileless malware hiding in RAM |
| **Sandboxing** | Isolates processes so memory corruption can't spread |

---

## 8. COMMON MISTAKES BEGINNERS MAKE

- Confusing **virtual memory (paging)** with **swap space** — swap is just virtual memory that uses disk
- Thinking **fileless malware** is impossible to detect — memory forensics catches it
- Not understanding that **ASLR and DEP work together** — bypassing one alone is usually not enough
- Confusing **stack** and **heap** — they serve different purposes and have different vulnerabilities

## 9. PRACTICAL LAB 🧪

**Objective:** Observe memory allocation and usage in real time

**On Windows:**
```powershell
# Check total and available RAM
Get-WmiObject Win32_OperatingSystem | 
Select-Object TotalVisibleMemorySize, FreePhysicalMemory

# Check virtual memory (pagefile)
Get-WmiObject Win32_PageFileUsage | 
Select-Object Name, CurrentUsage, PeakUsage

# Watch memory usage of a specific process
Get-Process chrome | Select-Object Name, WorkingSet, VirtualMemorySize
```

**On Linux:**
```bash
# Overall memory layout
free -h

# Detailed memory information
cat /proc/meminfo

# See memory map of a specific process (use any PID)
cat /proc/1/maps

# Watch memory usage in real time
watch -n 1 free -h

# Check swap usage
swapon --show
```

**Bonus — Simple Buffer Overflow Concept Demo:**
```python
# Run this Python script to understand buffer concept
# This is educational only

buffer_size = 8
user_input = "A" * 20  # 20 bytes into 8 byte buffer

if len(user_input) > buffer_size:
    print(f"[!] Buffer overflow detected!")
    print(f"    Buffer size: {buffer_size} bytes")
    print(f"    Input size:  {len(user_input)} bytes")
    print(f"    Overflow:    {len(user_input) - buffer_size} bytes past boundary")
else:
    print("[+] Input fits in buffer safely")
```

**Expected Output:**
```
[!] Buffer overflow detected!
    Buffer size: 8 bytes
    Input size:  20 bytes
    Overflow:    12 bytes past boundary
```

## 10. KEY TERMS

| Term | Simple Meaning |
|---|---|
| **Memory Management** | OS controlling how RAM is allocated and protected |
| **Stack** | Automatic memory for function calls, fixed size |
| **Heap** | Dynamic memory requested by programs manually |
| **Buffer** | Fixed size memory block reserved for data |
| **Buffer Overflow** | Writing more data than a buffer can hold |
| **Page** | Fixed size chunk of memory (usually 4KB) |
| **Page Fault** | Process needs a page that's been moved to disk |
| **MMU** | CPU component that translates virtual to physical addresses |
| **ASLR** | Randomizes memory addresses to prevent predictable exploitation |
| **DEP/NX** | Prevents data memory sections from being executed as code |
| **Stack Canary** | Special value detecting stack overflow attempts |
| **Heap Spray** | Filling heap with malicious code to increase exploit success |
| **Use-After-Free** | Bug where freed memory is still used — exploitable |
| **Fileless Malware** | Malware running entirely in RAM, never written to disk |
| **Volatility** | Industry standard memory forensics tool |
| **Memory Scraping** | Reading sensitive data directly from process memory |
