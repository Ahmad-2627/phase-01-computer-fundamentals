# Phase 1 — Computer Fundamentals
## Topic 1.10 — Users and Permissions

---

## 1. TOPIC OVERVIEW

Every system you will ever hack has one thing in common — it has **users** and those users have **permissions.**

Understanding users and permissions is critical because:
- Almost every attack involves gaining or abusing permissions
- Privilege escalation means moving from low permissions to high permissions
- Misconfigured permissions are one of the most common vulnerabilities in real organizations

This is your final topic in Phase 1 and it ties everything together.

---

## 2. CONCEPT EXPLANATION

**Users** are accounts that can log into and interact with a system.

**Permissions** are rules defining what each user is allowed to do — which files they can read, which programs they can run, which settings they can change.

> Simple analogy:
> Think of a large office building:
> - **Users** = Employees with ID cards
> - **Permissions** = Which doors their ID card can open
> - **Admin/Root** = Security officer whose card opens every door
> - **Guest** = Visitor who can only access the lobby
>
> A hacker's goal is often to get the security officer's card — or find a door that's been left unlocked by mistake.

## 3. INTERNAL WORKING

### 🔲 User Types — Linux

Linux has three main user categories:

| User Type | Description | Symbol |
|---|---|---|
| **Root** | Superuser — complete control over everything | # |
| **Regular User** | Normal account with limited permissions | $ |
| **Service Account** | Account that runs system services, not for humans | — |

**Root user:**
- UID (User ID) = **0** — always
- Can read, write, execute any file
- Can install software, change system settings
- Can create and delete other users

> **Why it matters for hacking:**
> Getting root = game over for the target system. Every privilege escalation attack aims for UID 0.

---

### 🔲 User Types — Windows

| User Type | Description |
|---|---|
| **Administrator** | Full control — equivalent to Linux root |
| **Standard User** | Limited permissions, can't install software system-wide |
| **Guest** | Minimal access, usually disabled |
| **SYSTEM** | Highest privilege — above Administrator, used by OS itself |

> **Why SYSTEM matters for hacking:**
> SYSTEM account is more powerful than Administrator. Many advanced exploits aim to get SYSTEM-level access — it's the Windows equivalent of root.

---

### 🔲 Linux File Permissions — The Core System

Every file and folder in Linux has three permission sets:

```
-rwxrwxrwx
│├──┤├──┤├──┤
││  ││  │└── Others (everyone else)
││  │└──┤─── Group (users in same group)
│└──┤────── Owner (user who owns file)
└────────── File type (- = file, d = directory)
```

Each set has three permission types:

| Symbol | Permission | Meaning |
|---|---|---|
| **r** | Read | Can view file contents |
| **w** | Write | Can modify file |
| **x** | Execute | Can run file as program |
| **-** | No permission | Access denied |

**Real example:**
```
-rwxr-xr--  file.sh
```

Breaking it down:
```
- = regular file
rwx = owner can read, write, execute
r-x = group can read and execute, NOT write
r-- = others can only read
```

---

### 🔲 Numeric (Octal) Permissions

Permissions also represented as numbers:

| Permission | Number |
|---|---|
| Read (r) | 4 |
| Write (w) | 2 |
| Execute (x) | 1 |
| No permission | 0 |

Add them together for each group:

| Combination | Calculation | Number |
|---|---|---|
| rwx | 4+2+1 | **7** |
| r-x | 4+0+1 | **5** |
| r-- | 4+0+0 | **4** |
| --- | 0+0+0 | **0** |

So `-rwxr-xr--` = **754**

**Common permission numbers:**

| Number | Meaning | Use Case |
|---|---|---|
| **777** | Everyone can do everything | Dangerous — never use on sensitive files |
| **755** | Owner full, others read+execute | Normal executables |
| **644** | Owner read+write, others read only | Normal files |
| **600** | Owner read+write only | Private files (SSH keys) |
| **400** | Owner read only | Very sensitive files |

---

### 🔲 Changing Permissions — chmod

```bash
# Symbolic method
chmod u+x file.sh      # Add execute for owner
chmod g-w file.sh      # Remove write from group
chmod o-r file.sh      # Remove read from others
chmod a+r file.sh      # Add read for all (a = all)

# Numeric method
chmod 755 file.sh      # rwxr-xr-x
chmod 644 file.txt     # rw-r--r--
chmod 600 id_rsa       # rw------- (SSH private key)
chmod 777 file.sh      # rwxrwxrwx (dangerous)
```

---

### 🔲 Changing Ownership — chown

```bash
# Change owner
chown ahmad file.txt

# Change owner and group
chown ahmad:developers file.txt

# Change ownership recursively (entire folder)
chown -R ahmad:developers /var/www/html
```

---

### 🔲 Special Permissions — SUID, SGID, Sticky Bit

These are advanced permission types that are heavily abused in privilege escalation.

**SUID — Set User ID:**

When set on an executable, it runs with the **file owner's permissions** instead of the running user's permissions.

```bash
# Finding SUID files (common priv esc technique)
find / -perm -4000 2>/dev/null
```

> **Why it matters for hacking:**
> If a file owned by **root** has SUID set — any user who runs it gets **root's permissions** temporarily. If that file has a vulnerability, attackers can exploit it for privilege escalation.
>
> Famous example: `/usr/bin/passwd` has SUID because it needs root to modify `/etc/shadow`.

**SGID — Set Group ID:**
Similar to SUID but applies group permissions instead.

**Sticky Bit:**
On directories — only the file owner can delete their own files even if others have write permission.

```bash
# Example: /tmp always has sticky bit
ls -la / | grep tmp
# drwxrwxrwt  ← t at end = sticky bit
```

---

### 🔲 Windows Permissions — ACL System

Windows uses **ACL (Access Control List)** — a list attached to every file defining who can do what.

Each entry in the ACL is called an **ACE (Access Control Entry).**

```
File: C:\Sensitive\passwords.txt
ACL:
├── SYSTEM → Full Control
├── Administrators → Full Control
├── Ahmad → Read
└── Guest → No Access
```

**Key Windows permission types:**

| Permission | Meaning |
|---|---|
| **Full Control** | Everything including changing permissions |
| **Modify** | Read, write, delete — but not change permissions |
| **Read & Execute** | View and run |
| **Read** | View only |
| **Write** | Create and modify — but not delete |
| **No Access** | Completely blocked |

---

### 🔲 Sudo — Running Commands as Root (Linux)

**sudo** = "superuser do" — run a single command as root

```bash
# Run command as root
sudo apt update

# Switch to root shell
sudo su

# Check what sudo commands you can run
sudo -l
```

> **Why it matters for hacking:**
> `sudo -l` is one of the first commands penetration testers run after getting access to a Linux system. It shows exactly which commands the current user can run as root — often revealing a privilege escalation path.

---

### 🔲 /etc/passwd and /etc/shadow

Two critical Linux files for user management:

**/etc/passwd** — readable by everyone:
```
root:x:0:0:root:/root:/bin/bash
ahmad:x:1001:1001::/home/ahmad:/bin/bash
```
Format: `username:password:UID:GID:info:home:shell`

The `x` means password is stored in /etc/shadow instead.

**/etc/shadow** — readable only by root:
```
root:$6$salt$hashedpassword:18000:0:99999:7:::
ahmad:$6$salt$hashedpassword:18500:0:99999:7:::
```

Contains actual **hashed passwords** — prime target for attackers.

---

## 4. WHY IT MATTERS

- **Every attack ends with permission abuse** — reading files you shouldn't, running code as admin
- **Misconfigured permissions** are the most common real-world vulnerability
- **SUID files** are a classic Linux privilege escalation vector
- **Weak ACLs on Windows** allow attackers to modify critical files
- **sudo misconfigurations** give attackers easy root access
- **/etc/shadow** is the target for offline password cracking

---

## 5. REAL-WORLD EXAMPLE

**Scenario — SUID Privilege Escalation**

A penetration tester gains low-privilege access to a Linux server.

```bash
# Step 1 — Check current user
whoami
# Output: www-data (web server user, low privilege)

# Step 2 — Find SUID files
find / -perm -4000 2>/dev/null
# Output: /usr/bin/find (owned by root, has SUID set!)

# Step 3 — Exploit SUID on 'find' binary
/usr/bin/find . -exec /bin/bash -p \;
# Output: bash-5.0# (root shell!)

whoami
# Output: root
```

This is a real privilege escalation technique found on GTFOBins — a resource you will use extensively in your pentesting career.

## 6. ATTACKER PERSPECTIVE 🔴

| Attack | Permission Involvement |
|---|---|
| **Privilege escalation** | Moving from low user to root/SYSTEM |
| **SUID abuse** | Exploiting SUID binaries owned by root |
| **Sudo misconfiguration** | Abusing overly permissive sudo rules |
| **Weak ACL abuse** | Modifying files attacker shouldn't have access to |
| **/etc/shadow theft** | Reading password hashes for offline cracking |
| **Token impersonation** | Stealing Windows user tokens for higher access |
| **Writable PATH abuse** | Placing malicious files in world-writable directories |

---

## 7. DEFENDER PERSPECTIVE 🔵

| Defense | What It Does |
|---|---|
| **Principle of least privilege** | Give users only minimum permissions needed |
| **Regular permission audits** | Check for misconfigured file permissions |
| **SUID auditing** | Remove unnecessary SUID bits from binaries |
| **Sudo hardening** | Restrict exactly which commands users can sudo |
| **Shadow file protection** | Ensure /etc/shadow is only root-readable |
| **ACL reviews** | Regularly audit Windows file permissions |
| **PAM (Pluggable Auth Modules)** | Enforce strong authentication policies on Linux |

---

## 8. COMMON MISTAKES BEGINNERS MAKE

- Setting **chmod 777** on files in their lab — dangerous habit to develop
- Running everything as **root on Kali** — builds bad habits, always use a regular user
- Ignoring **SUID files** during penetration tests — they're gold for privilege escalation
- Not checking **sudo -l** immediately after getting shell access
- Confusing **authentication** (proving who you are) with **authorization** (what you're allowed to do)

## 9. PRACTICAL LAB 🧪

**Objective:** Explore users, permissions, and practice privilege concepts

**On Linux:**
```bash
# Check current user and groups
whoami
id

# List all users on system
cat /etc/passwd

# Check your sudo permissions
sudo -l

# Create a test user (requires sudo/root)
sudo useradd testuser
sudo passwd testuser

# Switch to test user
su testuser

# Create a test file and check permissions
touch testfile.txt
ls -la testfile.txt
# Output: -rw-r--r-- 1 ahmad ahmad 0 Jun 1 testfile.txt

# Practice chmod
chmod 777 testfile.txt
ls -la testfile.txt
# Output: -rwxrwxrwx

chmod 600 testfile.txt
ls -la testfile.txt
# Output: -rw-------

# Find all SUID files (important pentesting skill)
find / -perm -4000 2>/dev/null

# Find world-writable files (misconfiguration hunting)
find / -perm -o+w -type f 2>/dev/null
```

**On Windows:**
```powershell
# Check current user
whoami

# Check user privileges
whoami /priv

# List all local users
net user

# List all local groups
net localgroup

# Check who is in Administrators group
net localgroup Administrators

# View file permissions
icacls C:\Windows\System32\cmd.exe

# Check your own group memberships
whoami /groups
```

**Key Learning Point:**
You just performed **basic privilege enumeration** — exactly what penetration testers do in the first minutes after gaining access to any system.
