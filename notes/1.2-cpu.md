# Phase 1 — Computer Fundamentals
## Topic 1.2 — CPU (The Brain of a Computer)

---

## 1. TOPIC OVERVIEW

You learned that CPU executes instructions. Now let's go deeper.

As a penetration tester you will run heavy tools like **Metasploit, Hashcat, Nmap.** All of these depend heavily on CPU. You also need to understand CPU to understand how **malware executes, how exploits work, and how programs run internally.**

---

## 2. CONCEPT EXPLANATION

CPU stands for **Central Processing Unit.**

Its only job is to **execute instructions** — one after another, billions of times per second.

Every program you run — whether it's a browser, a hacking tool, or malware — is just a list of instructions the CPU follows.

> Simple analogy:
> Think of CPU as a **chef** in a kitchen.
> - Recipe = Program (list of instructions)
> - Chef = CPU (follows instructions one by one)
> - Faster chef = Higher GHz CPU
> - Multiple chefs = Multiple cores

---

## 3. INTERNAL WORKING

### 🔲 How CPU Executes Instructions — The Fetch-Decode-Execute Cycle

Every instruction goes through 3 steps:

```
FETCH → DECODE → EXECUTE
```

**FETCH** — CPU gets the next instruction from memory

**DECODE** — CPU figures out what the instruction means

**EXECUTE** — CPU actually does the task

> Example:
> Instruction: "Add number 5 and number 3"
> - Fetch: Gets the instruction from RAM
> - Decode: Understands it means addition
> - Execute: Calculates 5+3 = 8

This happens **billions of times per second.**

---

### 🔲 CPU Speed — Clock Speed (GHz)

**GHz = Gigahertz** — how many cycles per second

| Speed | Meaning |
|---|---|
| 1 GHz | 1 billion cycles per second |
| 3.5 GHz | 3.5 billion cycles per second |

Higher GHz = faster execution = tools run faster

---

### 🔲 CPU Cores

A **core** is one independent processing unit inside the CPU.

| Type | Meaning |
|---|---|
| Single Core | 1 chef in the kitchen |
| Dual Core | 2 chefs working simultaneously |
| Quad Core | 4 chefs |
| 8 Core | 8 chefs |

More cores = more tasks handled at the same time

> **Why it matters for hacking:**
> Nmap scanning + Metasploit running + Burp Suite open
> = 3 tasks at once = needs multiple cores

---

### 🔲 CPU Registers

Registers are **tiny, ultra-fast storage** inside the CPU itself.

They temporarily hold:
- The instruction being executed
- Numbers being calculated
- Memory addresses

> Think of registers as the chef's **two hands** — they can only hold a small amount at one time but work extremely fast.

**Important registers you'll hear about in hacking:**

| Register | Purpose |
|---|---|
| **EIP / RIP** | Stores address of next instruction to execute |
| **ESP / RSP** | Points to top of the stack (memory area) |
| **EAX / RAX** | Stores results of calculations |

> **Why it matters for hacking:**
> **Buffer overflow attacks** work by overwriting the **EIP register** to make the CPU jump to malicious code instead of the normal next instruction. This is an advanced topic but now you understand why registers matter.

---

### 🔲 CPU Architecture — 32-bit vs 64-bit

| Type | Meaning |
|---|---|
| **32-bit (x86)** | Can handle data in 32-bit chunks, max 4GB RAM |
| **64-bit (x64)** | Can handle 64-bit chunks, supports much more RAM |

Most modern systems are **64-bit.**

> **Why it matters for hacking:**
> When you create exploits or run tools, you must know if the target is 32-bit or 64-bit. Wrong architecture = exploit fails.

---

### 🔲 Kernel Mode vs User Mode

CPU operates in two modes:

| Mode | Who Uses It | What It Can Do |
|---|---|---|
| **User Mode** | Normal programs, apps | Limited access to hardware |
| **Kernel Mode** | Operating system | Full access to everything |

> **Why it matters for hacking:**
> Many attacks try to **escalate from User Mode to Kernel Mode** — called **Privilege Escalation.** This gives attackers full control of the system.

---

## 4. WHY IT MATTERS

- Every exploit you run **executes on the CPU**
- Buffer overflow attacks **manipulate CPU registers**
- Password cracking speed **depends on CPU cores and GHz**
- Understanding 32-bit vs 64-bit **prevents tool failures**
- Privilege escalation targets the **boundary between user and kernel mode**

---

## 5. REAL-WORLD EXAMPLE

**Scenario:** You are cracking a password hash in an authorized lab.

- Tool: John the Ripper
- Your CPU: 4 cores, 3.2 GHz
- The tool uses all 4 cores simultaneously
- Each core tries millions of password combinations per second

Without understanding CPU cores, you wouldn't know why enabling multi-core mode in your tool makes it **4x faster.**

---

## 6. ATTACKER PERSPECTIVE 🔴

Attackers target CPU knowledge to:
- **Buffer overflow** — overwrite EIP register to redirect execution to malicious code
- **Privilege escalation** — move from user mode to kernel mode for full system control
- **Cryptojacking** — secretly use victim's CPU to mine cryptocurrency
- **Side-channel attacks** — read CPU behavior to steal encryption keys (Spectre/Meltdown vulnerabilities — famous real attacks)

---

## 7. DEFENDER PERSPECTIVE 🔵

Defenders protect CPU by:
- **Keeping OS updated** — patches fix CPU vulnerabilities like Spectre/Meltdown
- **DEP (Data Execution Prevention)** — stops CPU from executing malicious code in memory
- **ASLR (Address Space Layout Randomization)** — randomizes memory addresses so attackers can't predict where to redirect CPU execution
- **Monitoring CPU usage** — sudden 100% CPU usage can indicate cryptojacking malware

---

## 8. COMMON MISTAKES BEGINNERS MAKE

- Confusing **clock speed** with **cores** — a 4-core 2GHz CPU can be faster than a single-core 4GHz CPU for multi-tasking
- Allocating too little CPU to their **virtual machine** — causes tools to crash or run extremely slow
- Ignoring **32-bit vs 64-bit** when downloading tools or exploits

---

## 9. PRACTICAL LAB 🧪

**Objective:** Inspect CPU details on your own system

**On Windows:**
```
1. Press Ctrl + Shift + Esc (opens Task Manager)
2. Click "Performance" tab
3. Click "CPU"
4. Observe: Speed (GHz), Cores, and current usage %
```

**On Linux (Kali):**
```bash
# Full CPU details
lscpu

# Watch real-time CPU usage
top

# Or better view
htop
```

**What to look for:**
```
Architecture:    x86_64        ← 64-bit system
CPU(s):          4             ← Number of cores
Thread(s) per core: 2         ← Hyper-threading
CPU MHz:         3200          ← Current speed
```

**Expected Outcome:**
You now know your CPU specs. When you set up your hacking lab VM, you'll allocate the right number of cores.
