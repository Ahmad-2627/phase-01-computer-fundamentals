# Phase 1 — Computer Fundamentals
## Topic 1.8 — Processes (Programs in Action)

---

## 1. TOPIC OVERVIEW

You now know how CPU executes instructions, how RAM stores running programs, and how file systems organize files on disk.

Now the question is — **what exactly is a running program called?**

A **Process.**

Processes are one of the most important concepts in penetration testing. Malware runs as a process. Passwords are extracted from processes. Attackers inject code into processes. If you understand processes deeply, you understand how attacks work at their core.

---

## 2. CONCEPT EXPLANATION

A **process** is a program that is currently loaded into RAM and being executed by the CPU.

Simple distinction:

| Term | Meaning |
|---|---|
| **Program** | A file sitting on storage (inactive) |
| **Process** | That same program loaded into RAM and running (active) |

> Simple analogy:
> - **Program** = A recipe written in a cookbook (sitting on shelf, doing nothing)
> - **Process** = A chef actively cooking that recipe right now (using ingredients, taking up counter space)
>
> You can have one recipe but run it as multiple processes simultaneously — like two chefs cooking the same dish at the same time.

---

## 3. INTERNAL WORKING

### 🔲 What a Process Contains

When a program becomes a process, the OS creates a complete environment for it in RAM:

```
┌─────────────────────────────┐
│          Process            │
├─────────────────────────────┤
│  PID (Process ID)           │ ← Unique number identifying this process
│  Code Section               │ ← Program instructions
│  Data Section               │ ← Global variables
│  Heap                       │ ← Dynamic memory
│  Stack                      │ ← Function calls and local variables
│  File handles               │ ← Files this process has open
│  Network connections        │ ← Open network connections
│  Security token             │ ← Permissions this process runs with
└─────────────────────────────┘
```

---

### 🔲 PID — Process ID

Every process gets a unique number called a **PID (Process ID).**

The OS uses PIDs to track and manage processes.

| PID | Example Process |
|---|---|
| 1 | systemd (Linux — first process ever) |
| 4 | System (Windows — core kernel process) |
| 1337 | Your browser (example) |
| 4444 | Could be anything — number is dynamic |

> **Why it matters for hacking:**
> When you want to inject code into a process or kill a malicious process, you target it by PID.

---

### 🔲 Process States

A process doesn't always use CPU. It switches between states:

```
NEW → READY → RUNNING → WAITING → TERMINATED
```

| State | Meaning |
|---|---|
| **New** | Process just created |
| **Ready** | Waiting for CPU time |
| **Running** | Currently executing on CPU |
| **Waiting** | Paused, waiting for something (file read, network) |
| **Terminated** | Process finished and being removed from RAM |

> The OS switches between thousands of processes per second — this is called **scheduling.** It happens so fast it feels like everything runs simultaneously.

---

### 🔲 Parent and Child Processes

Processes can create other processes.

| Term | Meaning |
|---|---|
| **Parent Process** | Process that created another process |
| **Child Process** | Process created by a parent |

```
systemd (PID 1) — Parent
└── sshd (PID 120) — Child (SSH service)
    └── bash (PID 543) — Child of sshd (your terminal session)
        └── nmap (PID 891) — Child of bash (tool you ran)
```

> **Why it matters for hacking:**
> When you exploit a web server and get a shell, your shell is a **child process** of the web server process. Security tools look for **suspicious parent-child relationships** — like a web server spawning a command shell, which is abnormal.

---

### 🔲 Threads

A **thread** is a smaller unit of execution inside a process.

| Term | Meaning |
|---|---|
| **Process** | Complete program with its own memory space |
| **Thread** | Lightweight task running inside a process |

One process can have many threads running simultaneously.

> Example: Your browser is one process but has many threads — one loading a page, one playing video, one checking downloads.

> **Why it matters for hacking:**
> Some malware uses multiple threads to perform different malicious tasks simultaneously — downloading files, keylogging, and communicating with attackers all at once.

---

### 🔲 Process Memory Space

Each process gets its own **isolated memory space** in RAM.

```
Process A Memory    Process B Memory
┌────────────┐      ┌────────────┐
│ Code       │      │ Code       │
│ Heap       │      │ Heap       │
│ Stack      │      │ Stack      │
└────────────┘      └────────────┘
     ↑                    ↑
  Isolated            Isolated
```

Processes cannot normally read each other's memory — the OS enforces this.

> **Why it matters for hacking:**
> **Process injection** is an attack that breaks this isolation — injecting malicious code into a legitimate process's memory space. This lets malware hide inside trusted processes like explorer.exe or svchost.exe.

---

### 🔲 Important Windows Processes

Knowing normal processes helps you spot malicious ones.

| Process | Purpose | Why Hackers Care |
|---|---|---|
| **System (PID 4)** | Core kernel operations | Targeted by rootkits |
| **lsass.exe** | Stores login credentials | Mimikatz targets this |
| **svchost.exe** | Hosts Windows services | Malware disguises as this |
| **explorer.exe** | Windows desktop/GUI | Common injection target |
| **winlogon.exe** | Handles login screen | Credential theft target |
| **cmd.exe** | Command prompt | Spawned by exploits |
| **powershell.exe** | PowerShell | Heavily abused by malware |

> **Red flag:** If you see `svchost.exe` running from `C:\Temp\` instead of `C:\Windows\System32\` — that's malware. Legitimate svchost.exe only runs from System32.

---

### 🔲 Important Linux Processes

| Process | Purpose |
|---|---|
| **systemd (PID 1)** | First process, parent of everything |
| **sshd** | SSH service for remote access |
| **apache2 / nginx** | Web server processes |
| **bash / sh** | Command shell |
| **cron** | Scheduled task runner |
| **ps** | Lists running processes |

---

### 🔲 Process Permissions and Tokens (Windows)

In Windows, every process runs with a **security token** defining:
- Which user it belongs to
- What privileges it has
- What resources it can access

> **Why it matters for hacking:**
> **Token impersonation** is an attack where a penetration tester steals the security token from a high-privilege process and uses it to run commands as Administrator or SYSTEM — without knowing the password.

---

## 4. WHY IT MATTERS

- **Malware identification** — spotting suspicious processes on a compromised system
- **Process injection** — injecting malicious code into legitimate processes
- **Credential theft** — extracting passwords from lsass.exe process
- **Privilege escalation** — abusing process tokens for higher permissions
- **Post-exploitation** — first thing you do after compromise is list running processes
- **Persistence** — malware registers itself to start as a process on boot

---

## 5. REAL-WORLD EXAMPLE

**Scenario — Process Injection Attack**

An attacker compromises a Windows machine but their malware keeps getting detected by antivirus.

**Solution — Process Injection:**

```
1. Attacker identifies explorer.exe running (trusted Windows process)
2. Attacker injects malicious shellcode into explorer.exe memory
3. Malicious code now runs INSIDE explorer.exe
4. Antivirus sees only explorer.exe — a trusted process
5. Malicious activity is hidden inside a legitimate process
6. Network connections appear to come from explorer.exe — trusted
```

This is why modern **EDR (Endpoint Detection and Response)** tools monitor process behavior, not just process names.

---

## 6. ATTACKER PERSPECTIVE 🔴

| Attack | Process Involvement |
|---|---|
| **Process injection** | Inject malicious code into legitimate process memory |
| **Process hollowing** | Empty a legitimate process and replace with malware |
| **LSASS dumping** | Read credentials from lsass.exe process memory |
| **Token impersonation** | Steal high-privilege process token |
| **Malicious child process** | Exploit spawns cmd.exe or powershell.exe as child |
| **Cron job abuse (Linux)** | Add malicious scheduled process for persistence |
| **Rootkit** | Hide malicious processes from process listing tools |

---

## 7. DEFENDER PERSPECTIVE 🔵

| Defense | What It Does |
|---|---|
| **EDR monitoring** | Watches process behavior and relationships |
| **Process whitelisting** | Only allow approved processes to run |
| **Credential Guard** | Protects lsass.exe from memory dumping |
| **PowerShell logging** | Records all PowerShell process activity |
| **Parent process monitoring** | Alerts on suspicious parent-child relationships |
| **Memory scanning** | Detects injected code in process memory |

---

## 8. COMMON MISTAKES BEGINNERS MAKE

- Confusing **program** and **process** — a program becomes a process only when running
- Not checking **process location** — same name, different path = likely malware
- Killing the wrong process in lab — always double check PID before killing
- Ignoring **parent-child relationships** — they reveal how attacks happened

---

## 9. PRACTICAL LAB 🧪

**Objective:** List, analyze, and understand running processes on your system

**On Windows:**
```powershell
# List all running processes
tasklist

# Detailed process list with memory usage
tasklist /v

# Find specific process
tasklist | findstr "lsass"

# List processes with their full path (PowerShell)
Get-Process | Select-Object Name, ID, Path

# Kill a process by PID (be careful)
taskkill /PID [PID_NUMBER] /F

# See parent-child relationships
Get-WmiObject Win32_Process | Select-Object Name, ProcessId, ParentProcessId
```

**On Linux:**
```bash
# List all processes
ps aux

# Tree view showing parent-child relationships
pstree

# Interactive real-time process viewer
top
# or better:
htop

# Find specific process
ps aux | grep ssh

# Get PID of a process by name
pidof sshd

# Kill a process by PID
kill [PID]

# Kill by name
killall processname

# See open files by a process
lsof -p [PID]

# See network connections by process
ss -tulnp
```

**Investigation Exercise:**
```bash
# On Linux — find all processes running as root
ps aux | grep root

# Find processes listening on network ports
ss -tulnp

# Check if any process is connecting to internet
netstat -tulnp
```

**Expected Outcome:**
- You can list all running processes
- You can identify process names, PIDs, and owners
- You can spot which processes have network connections
- You understand parent-child relationships

**Key Learning Point:**
Listing processes and checking network connections is called **situational awareness** — the first thing penetration testers do after gaining access to a system.

| Term | Simple Meaning |
|---|---|
