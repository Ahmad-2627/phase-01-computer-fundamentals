# Phase 1 — Computer Fundamentals
## Topic 1.3 — RAM (Temporary Memory)

---

## 1. TOPIC OVERVIEW

You now know CPU executes instructions. But where does it get those instructions from?

**RAM.**

RAM is one of the most important concepts in hacking. Malware hides in RAM. Passwords live in RAM. Encryption keys sit in RAM. If you understand RAM, you understand why many powerful attacks work.

---

## 2. CONCEPT EXPLANATION

RAM stands for **Random Access Memory.**

It is your computer's **short-term memory** — fast, temporary workspace.

When you open a program:
1. It loads from Storage (HDD/SSD) into RAM
2. CPU reads and executes it from RAM
3. When you close the program or shut down — RAM is wiped clean

> Simple analogy:
> - **Storage (HDD/SSD)** = Your bookshelf (permanent, stores everything)
> - **RAM** = Your study desk (temporary, only what you're currently working on)
> - **CPU** = Your brain (reads from the desk, not the bookshelf directly)

The desk gets cleared every time you leave the room (shutdown).

---

## 3. INTERNAL WORKING

### 🔲 How RAM Works With CPU

```
Program on Storage
       ↓
Loaded into RAM
       ↓
CPU fetches instructions from RAM
       ↓
CPU executes them
       ↓
Results stored back in RAM temporarily
```

CPU cannot directly read from HDD/SSD — it's too slow. RAM is the middle layer.

---

### 🔲 RAM Speed vs Storage Speed

| Type | Speed |
|---|---|
| HDD | ~100 MB/s |
| SSD | ~500 MB/s |
| RAM | ~30,000 MB/s |
| CPU Cache | ~300,000 MB/s |

RAM is **300x faster than HDD.** That's why programs must load into RAM before running.

---

### 🔲 RAM Capacity

Measured in **GB (Gigabytes)**

| RAM Size | Suitable For |
|---|---|
| 4GB | Basic use only |
| 8GB | Minimum for hacking lab |
| 16GB | Comfortable for multiple VMs |
| 32GB | Professional lab setup |

> **Important for your lab:**
> Running Kali Linux + Windows VM simultaneously needs at least **8GB RAM** on your physical machine.

---

### 🔲 What Lives Inside RAM

This is where it gets interesting for hacking.

At any moment, RAM contains:

| What | Why It Matters to Hackers |
|---|---|
| **Running programs** | Malware runs here |
| **Passwords** | Plaintext passwords exist briefly in RAM |
| **Encryption keys** | Keys must be in RAM to work |
| **Session tokens** | Used to hijack logged-in sessions |
| **Clipboard data** | Whatever you copied is here |
| **Browser data** | Open tabs, form data |
| **OS processes** | Core system functions |

---

### 🔲 Memory Layout — How RAM is Organized

RAM is divided into sections for each running program:

```
HIGH ADDRESS
┌─────────────────┐
│   Stack         │ ← Local variables, function calls
├─────────────────┤
│   Heap          │ ← Dynamically allocated memory
├─────────────────┤
│   Data          │ ← Global variables
├─────────────────┤
│   Code (Text)   │ ← Program instructions
└─────────────────┘
LOW ADDRESS
```

**Stack** — temporary storage for active functions
**Heap** — memory programs request while running
**Code section** — actual program instructions CPU executes

> **Why it matters for hacking:**
> **Buffer overflow attacks** happen when too much data is written into the **Stack**, overflowing into other areas and overwriting the EIP register you learned about in Topic 1.2.

---

### 🔲 Virtual Memory

When RAM is full, the OS uses a section of your **HDD/SSD as fake RAM.**

This is called **Virtual Memory** or **Page File (Windows) / Swap (Linux).**

It's much slower than real RAM but prevents crashes.

> **Why it matters for hacking:**
> Sensitive data moved to virtual memory (pagefile) on disk can sometimes be **recovered by attackers** even after the program closes.

---

## 4. WHY IT MATTERS

- **Malware analysis** — malware runs in RAM, analysts dump RAM to study it
- **Password attacks** — tools like Mimikatz extract passwords directly from RAM
- **Buffer overflow exploits** — manipulate stack and heap in RAM
- **Session hijacking** — steal session tokens stored in RAM/browser memory
- **Cold boot attacks** — RAM retains data briefly after shutdown, attackers freeze and extract it

---

## 5. REAL-WORLD EXAMPLE

**Real Attack — Mimikatz**

Mimikatz is a famous tool used by penetration testers (and attackers) in authorized assessments.

What it does:
- Accesses the **LSASS process** in RAM
- LSASS = Local Security Authority Subsystem Service
- Windows stores **login credentials temporarily** in this process
- Mimikatz dumps those credentials **directly from RAM**

Result: Attacker gets plaintext passwords or password hashes without ever touching the hard drive.

This is why defenders now configure Windows to **not store plaintext passwords in LSASS.**

---

## 6. ATTACKER PERSPECTIVE 🔴

| Attack | How RAM is Involved |
|---|---|
| **Mimikatz** | Dumps passwords from RAM process |
| **Buffer Overflow** | Overflows stack in RAM to hijack execution |
| **Cold Boot Attack** | Extracts encryption keys from RAM after shutdown |
| **Heap Spray** | Fills heap with malicious code |
| **Process Injection** | Injects malicious code into legitimate RAM process |
| **Pagefile Analysis** | Recovers sensitive data from virtual memory on disk |

---

## 7. DEFENDER PERSPECTIVE 🔵

| Defense | What It Does |
|---|---|
| **Credential Guard (Windows)** | Protects LSASS from Mimikatz-style attacks |
| **Full Disk Encryption** | Protects pagefile data on disk |
| **DEP** | Prevents code execution in data sections of RAM |
| **ASLR** | Randomizes memory addresses so attackers can't predict locations |
| **Secure Boot** | Ensures nothing malicious loads into RAM at startup |
| **RAM Encryption** | Some modern systems encrypt RAM itself |

---

## 8. COMMON MISTAKES BEGINNERS MAKE

- Thinking deleted files are gone from RAM immediately — data lingers briefly
- Allocating too little RAM to VMs — 2GB for Kali Linux is not enough, use minimum 4GB
- Confusing **Virtual Memory** with **RAM** — virtual memory is just slow disk space pretending to be RAM
- Not understanding why tools crash — usually RAM is full

## 9. PRACTICAL LAB 🧪

**Objective:** Observe what is actually running in your RAM right now

**On Windows:**
```
1. Press Ctrl + Shift + Esc
2. Click "Processes" tab
3. Click "Memory" column to sort by RAM usage
4. Observe which programs are using the most RAM
5. Click "Performance" → "Memory" to see total usage
```

**On Linux:**
```bash
# See RAM usage summary
free -h

# See processes using most RAM
top
# Then press Shift + M to sort by memory

# Detailed memory info
cat /proc/meminfo
```

**What to observe:**
```
Total RAM:     8GB
Used:          4.2GB
Available:     3.8GB
```

Look at running processes and notice:
- Your browser uses a lot of RAM
- System processes are always running
- Available RAM decreases as you open more programs

**Key Learning Point:**
You just listed all **active processes in memory** — this is exactly what a penetration tester does when checking what's running on a compromised system.

## 10. KEY TERMS

| Term | Simple Meaning |
|---|---|
| **RAM** | Fast temporary memory, wiped on shutdown |
| **Stack** | RAM section for active function data |
| **Heap** | RAM section for dynamically requested memory |
| **Virtual Memory** | Disk space used as slow backup RAM |
| **Pagefile / Swap** | Windows/Linux name for virtual memory file |
| **LSASS** | Windows process that holds login credentials in RAM |
| **Mimikatz** | Tool that extracts passwords from RAM |
| **Buffer Overflow** | Attack overflowing stack memory to hijack CPU |
| **Cold Boot Attack** | Extracting RAM data briefly after shutdown |
| **Process Injection** | Hiding malicious code inside legitimate RAM process |
| **DEP** | Prevents malicious execution in RAM data areas |
| **ASLR** | Randomizes RAM addresses to confuse attackers |
