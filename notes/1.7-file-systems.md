# Phase 1 — Computer Fundamentals
## Topic 1.7 — File Systems (How Data is Stored)

---

## 1. TOPIC OVERVIEW

You learned that storage keeps data permanently. But how does the OS actually **organize** that data?

That's the job of the **File System.**

As a penetration tester you will navigate file systems on target machines, find sensitive files, recover deleted data, and understand where attackers hide malicious files. This topic is essential.

---

## 2. CONCEPT EXPLANATION

A **File System** is a method the OS uses to:
- Organize files into folders
- Track where each file is stored on disk
- Manage file names, sizes, and dates
- Control who can access which files

Without a file system, your storage would just be billions of random 0s and 1s with no organization.

> Simple analogy:
> Imagine a huge library with millions of books thrown randomly on the floor.
> - **Without file system** = Books thrown everywhere, impossible to find anything
> - **With file system** = Books organized by shelves, categories, labels, and a catalog
>
> The file system is the **library catalog + organization system.**

---

## 3. INTERNAL WORKING

### 🔲 How Files Are Actually Stored

Storage is divided into tiny fixed-size blocks called **clusters** or **blocks.**

```
Physical Storage Disk
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ B1 │ B2 │ B3 │ B4 │ B5 │ B6 │ B7 │ B8 │
└────┴────┴────┴────┴────┴────┴────┴────┘
```

A file can be split across multiple blocks — called **fragmentation.**

The file system keeps a **table** tracking which blocks belong to which file.

---

### 🔲 Key File Systems Explained

---

**NTFS — New Technology File System (Windows)**

Used by: Windows XP, 7, 8, 10, 11, Windows Server

Key features:

| Feature | Meaning |
|---|---|
| **Journaling** | Records changes before making them — recovers from crashes |
| **Permissions** | Controls who reads/writes/executes each file |
| **Encryption (EFS)** | Can encrypt individual files |
| **Compression** | Can compress files to save space |
| **Alternate Data Streams (ADS)** | Hidden data attached to files |
| **Max file size** | 16TB |

> **Why ADS matters for hacking:**
> Attackers hide malware inside **Alternate Data Streams** — invisible to normal file browsing but executable. This is a real attack technique.

---

**FAT32 — File Allocation Table (Legacy)**

Used by: USB drives, older systems, memory cards

| Feature | Detail |
|---|---|
| **Max file size** | Only 4GB — huge limitation |
| **Compatibility** | Works on almost every device |
| **Security** | No permissions — anyone can read everything |
| **Journaling** | No — data loss risk on crash |

> **Why it matters for hacking:**
> USB drives use FAT32. No permissions = no access control = easy data theft from USB.

---

**ext4 — Fourth Extended File System (Linux)**

Used by: Ubuntu, Kali Linux, Debian, most Linux systems

| Feature | Detail |
|---|---|
| **Journaling** | Yes — crash recovery |
| **Permissions** | Strong Unix-style permissions |
| **Max file size** | 16TB |
| **Performance** | Fast and reliable |

> This is the file system your Kali Linux will use.

---

**exFAT — Extended FAT**

Used by: Modern USB drives, SD cards

| Feature | Detail |
|---|---|
| **Max file size** | No practical limit |
| **Compatibility** | Works on Windows, Mac, Linux |
| **Security** | No permissions |

### 🔲 File System Structure — How Files Are Organized

**Windows Structure:**
```
C:\
├── Windows\          ← OS files
├── Program Files\    ← Installed applications
├── Users\
│   └── Ahmad\
│       ├── Desktop\
│       ├── Documents\
│       ├── Downloads\
│       └── AppData\  ← Hidden — important for malware
└── Temp\             ← Temporary files — malware loves this
```

**Linux Structure:**
```
/
├── bin\      ← Essential commands (ls, cat, pwd)
├── etc\      ← Configuration files — very important
├── home\     ← User home directories
│   └── ahmad\
├── var\      ← Logs and variable data
│   └── log\  ← System logs — critical for investigation
├── tmp\      ← Temporary files — malware loves this
├── root\     ← Root user's home directory
├── usr\      ← User programs and utilities
└── proc\     ← Virtual — running process information
```

> **Critical paths for penetration testing:**

| Path | Why It Matters |
|---|---|
| **C:\Windows\System32** | Core Windows files — malware hides here |
| **C:\Users\[user]\AppData** | Hidden folder — malware stores configs here |
| **C:\Temp** | Common malware drop location |
| **/etc/passwd** | Linux user account information |
| **/etc/shadow** | Linux password hashes — prime target |
| **/var/log/** | System logs — evidence of attacks |
| **/tmp** | Common malware execution location |

### 🔲 File Metadata

Every file has **metadata** — data about the file itself.

| Metadata | Meaning |
|---|---|
| **Name** | File name |
| **Size** | How large the file is |
| **Created** | When file was first created |
| **Modified** | When file content was last changed |
| **Accessed** | When file was last opened |
| **Permissions** | Who can read/write/execute |
| **Owner** | Which user owns the file |

> **Why it matters for hacking:**
> Forensic investigators use **timestamps** to reconstruct exactly what happened during an attack. Attackers sometimes modify timestamps to hide their activity — called **timestomping.**

---

### 🔲 Alternate Data Streams (ADS) — Windows Only

This is a hidden NTFS feature that attackers abuse.

Every NTFS file has:
- **Main stream** = Normal visible file content
- **Alternate streams** = Hidden extra data attached to the same file

```bash
# Creating a hidden ADS (educational example)
echo "hidden malware" > normal_file.txt:hidden_stream

# Normal file looks empty/normal
# But hidden stream contains data
# Most antivirus tools miss this
```

> Defenders use tools like **Streams.exe** or **PowerShell** to detect ADS.

---

### 🔲 Inodes (Linux Only)

In Linux, every file has an **inode** — a data structure storing all metadata.

```
Inode contains:
- File size
- Owner
- Permissions
- Timestamps
- Pointers to data blocks
```

Importantly — the inode does **not** store the file name. The directory stores the name → inode mapping.

> **Why it matters for hacking:**
> When a file is "deleted" in Linux, only the directory entry is removed. The inode and data blocks remain until overwritten — enabling file recovery.

---

## 4. WHY IT MATTERS

- Finding **sensitive files** on target systems requires understanding file structure
- **/etc/shadow** contains Linux password hashes — a prime target
- **AppData and Temp folders** are common malware locations
- **Alternate Data Streams** are used to hide malicious files
- **Timestamps** help forensic investigators and are tampered with by attackers
- **Deleted file recovery** depends on how the file system handles deletion

---

## 5. REAL-WORLD EXAMPLE

**Scenario — Attacker Hides Malware Using ADS**

An attacker gains access to a Windows machine and wants to hide their tools.

```
Normal approach: Drop malware.exe → Antivirus detects it

Smart approach:
1. Find an innocent looking file: readme.txt
2. Hide malware inside ADS: readme.txt:update.exe
3. Execute hidden stream
4. readme.txt looks completely normal
5. Many antivirus tools miss the hidden stream
```

This is a real technique used in **advanced persistent threat (APT)** attacks.

## 6. ATTACKER PERSPECTIVE 🔴

| Attack | File System Involvement |
|---|---|
| **Credential theft** | Reading /etc/shadow or Windows SAM file |
| **Log deletion** | Deleting /var/log files to hide activity |
| **ADS abuse** | Hiding malware in alternate data streams |
| **Timestomping** | Modifying file timestamps to confuse forensics |
| **Temp folder abuse** | Dropping and executing malware in /tmp or C:\Temp |
| **Deleted file recovery** | Recovering sensitive files victim deleted |
| **AppData persistence** | Storing malware configs in hidden AppData folder |

---

## 7. DEFENDER PERSPECTIVE 🔵

| Defense | What It Does |
|---|---|
| **File integrity monitoring** | Alerts when important files are modified |
| **Log protection** | Send logs to remote server immediately |
| **ADS scanning** | Regularly scan for alternate data streams |
| **Proper permissions** | Restrict who can access sensitive files |
| **Forensic imaging** | Take snapshots of file systems for investigation |
| **Immutable logs** | Logs that cannot be modified or deleted |

---

## 8. COMMON MISTAKES BEGINNERS MAKE

- Not knowing **where to look** on a target system — knowing key paths saves huge time
- Ignoring **hidden files** — both Windows and Linux hide important files by default
- Forgetting that **/etc/shadow** needs root access — reminder to always check privileges first
- Confusing **file system** with **operating system** — file system is just one component of OS

## 9. PRACTICAL LAB 🧪

**Objective:** Navigate file systems and find important security-related files

**On Windows:**
```powershell
# Show hidden files via PowerShell
Get-ChildItem -Hidden C:\Users\$env:USERNAME\AppData

# Check Temp folder
dir C:\Temp
dir C:\Windows\Temp

# Find ADS on a file (create test first)
echo "normal content" > test.txt
echo "hidden content" > test.txt:hidden
dir /r test.txt
```

**On Linux:**
```bash
# Navigate to important directories
ls /etc/
cat /etc/passwd        # User accounts
sudo cat /etc/shadow   # Password hashes (needs root)

# Check logs
ls /var/log/
cat /var/log/auth.log  # Login attempts

# Find hidden files (start with dot)
ls -la /home/         # -a shows hidden files

# Check temp directory
ls /tmp/

# Find files by name
find / -name "*.log" 2>/dev/null

# Find files modified in last 24 hours
find / -mtime -1 2>/dev/null
```

**Expected Outcome:**
- You can navigate both Windows and Linux file structures
- You found where logs, passwords, and configs live
- You understand which locations matter during penetration testing

## 10. KEY TERMS

| Term | Simple Meaning |
|---|---|
